# Sprint Documentation Standard

Use this format for every sprint going forward.

## 1) Sprint Start Plan (Locked)
- Sprint ID / date range
- Sprint goal (1-2 sentences)
- Planned work items (numbered, concrete, testable)

## 2) Epic Status
For each epic:
- Status: Planned | In Progress | Completed | Blocked
- Scope summary (what was intended)
- Acceptance criteria

## 3) Sprint End Report

### Done
- What was delivered (flat bullet list)
- Include meaningful implementation detail, not just filenames

### Validation
- Commands executed
- Pass/fail result for each command
- Note if verification is partial (focused tests only, no full regression, etc.)

### Risks
- Remaining technical risks after sprint completion
- Any rollout or stability concerns

### Learnings
- 2-5 concrete lessons from execution (engineering/process/product)

### Needs From Others
- Specific asks (data, decisions, approvals, access)
- Include owner when known

### Next Sprint Candidates
- Prioritized list (top 3 max)
- Each item should map to a measurable outcome

## 4) Naming Convention
- File pattern: `SPRINT_##_TASK_BOARD.md`
- Keep one sprint file per sprint.
- Keep this standards file unchanged except when process itself improves.
